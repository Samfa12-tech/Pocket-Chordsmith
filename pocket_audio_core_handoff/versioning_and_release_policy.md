# Pocket Audio Core Versioning and Release Policy

## 1. Purpose

Pocket Audio Core exists so the Pocket Chordsmith family has one shared audio truth.

Every Pocket Audio Core update should be treated as a shared engine release. A change to the core may affect:

- Pocket Chordsmith
- Pocket DJ
- Pocket DAW
- Godot export workflow
- new HTML/Three/Babylon games
- future Android/Capacitor builds

Do not silently update one consumer and leave the others drifting.

---

## 2. Version numbers

Use semantic versioning for the core:

```text
Pocket Audio Core 0.1.0
Pocket Audio Core 0.2.0
Pocket Audio Core 1.0.0
```

Keep project schema versions separate:

```text
Pocket Chordsmith project schema: 16, 17, 18...
Pocket Audio Core version: 0.1.0, 0.2.0, 1.0.0...
Pocket DJ session version: PDJ1, PDJ2...
Pocket DAW project version: PDAW1, PDAW2...
```

The audio core version describes the playback/rendering engine. The project schema version describes saved project data.

---

## 3. Compatibility matrix

Every core release must include a compatibility matrix like this:

```text
Pocket Audio Core: 0.2.0
Supports PCS project schemas: 16-17
Supports PCS share prefix: PCS1
Supports PDJ session prefix: PDJ1 read-only / full
Used by Pocket Chordsmith: vXX
Used by Pocket DJ: vYY
Used by Pocket DAW: vZZ
Godot exporter compatible: addon vAA.BB
Golden fixtures passing: yes/no
Known parity limitations: ...
```

---

## 4. Release rule

A core version is not considered released until:

1. Core tests pass.
2. Golden fixture event traces pass.
3. Offline render tests pass.
4. Pocket Chordsmith smoke test passes.
5. Pocket DJ smoke test passes.
6. Pocket DAW import/render smoke test passes once DAW exists.
7. Godot export/parity workflow smoke test passes once integrated.
8. Changelog is updated.
9. Compatibility matrix is updated.

---

## 5. App update rule

Whenever Pocket Audio Core changes, create a small app update even if the visible UI does not change.

Example:

```text
Pocket Chordsmith v70
- Updated Pocket Audio Core to 0.4.0.
- No project schema change.
- Audio parity tests pass with Pocket DJ v1h and Pocket DAW v0.2.
```

```text
Pocket DJ v1h
- Updated Pocket Audio Core to 0.4.0.
- Imports PCS1 projects using the same normaliser and playback engine as Pocket Chordsmith v70.
```

---

## 6. Exact-sound policy

For web apps, exact sound means the same core version and the same render/playback path.

For Godot, exact sound means exported audio assets rendered by the core, or event/sample manifests generated by the core. A separate native procedural preview can be useful but should be labelled as a preview if it is not parity-guaranteed.

---

## 7. Branching approach

Suggested branches:

```text
pocket-audio-core-design
pocket-audio-core-v0
pocket-audio-core-parity
pocket-dj-core-integration
pocket-chordsmith-core-integration
pocket-daw-core-foundation
godot-core-export-parity
```

Do not mix broad app rewrites with the first core extraction.

---

## 8. Changelog requirements

Every release note must say:

- core version
- project schema support
- new audio features
- changed instruments/FX
- changed scheduler behaviour
- changed offline render behaviour
- known parity limitations
- tests run
- apps updated

---

## 9. Deprecation policy

Do not delete old project fields without a migration.

If a field becomes legacy:

1. Keep reading it.
2. Normalise it into the new runtime model.
3. Export the new model only when a schema bump is explicit.
4. Document the migration.

---

## 10. Emergency rollback

Any app should be able to pin the previous core build if a serious regression appears.

Keep previous core distributions in an archive:

```text
core-releases/0.1.0/
core-releases/0.2.0/
core-releases/0.3.0/
```

A rollback should not change saved user projects unless a schema migration has already occurred.
