# Codex Prompt 07 — Godot Export Parity Workflow

Design and implement a Godot-facing workflow that treats Pocket Audio Core as the source of sound truth.

## Goal

When a Pocket Chordsmith song is exported to Godot, it should sound the same as Pocket Chordsmith where parity matters.

The safest route is not to rewrite Web Audio inside Godot. The safest route is to export audio assets and event manifests generated by Pocket Audio Core.

## Existing principle

Godot can continue using its native audio system and conductor node for runtime playback, transitions, ducking, buses and signals.

Pocket Audio Core should provide:

- rendered stems
- rendered loops
- rendered stingers
- generated drum/accent sample kits
- event manifests/timelines
- metadata for sections, BPM, bars, beat positions and loop points

## Required deliverables

Create or update:

```text
docs/GODOT_PARITY_EXPORT_WORKFLOW.md
packages/pocket-audio-core/src/export/godot-kit.js
packages/pocket-audio-core/examples/godot-export-demo/README.md
```

If working directly in the Godot addon repo, update appropriate addon docs and import tools, but do not break existing chart import.

## Export profiles

Support these conceptual profiles:

### 1. STEM_SYNC

Core renders each stem for a section/sequence:

```text
drums.wav
bass.wav
chords.wav
melody.wav
guitar.wav
full_mix.wav
manifest.json
```

Godot plays stems using native players/buses, keeping them aligned.

### 2. LOOP_KIT

Core renders loopable section mixes and optional stems:

```text
section_A_mix.wav
section_B_mix.wav
section_C_mix.wav
section_D_mix.wav
section_A_drums.wav
...
manifest.json
```

Godot conductor handles transitions between sections.

### 3. HYBRID

Core renders a main bed/stems and separate short accents/stingers/samples:

```text
bed_drums.wav
bed_bass.wav
bed_chords.wav
kick.wav
snare.wav
crash.wav
victory_stinger.wav
manifest.json
```

### 4. PROCEDURAL_PREVIEW

Godot-native procedural preview is allowed for editor convenience only. It must be labelled as preview unless parity is actually proven.

## Manifest

Design a manifest like:

```json
{
  "app": "PocketAudioCoreGodotKit",
  "coreVersion": "0.1.0",
  "sourceProjectSchema": 16,
  "bpm": 118,
  "timeSig": 4,
  "swing": 0.04,
  "sampleRate": 48000,
  "sections": {
    "A": {
      "bars": 4,
      "duration": 8.135,
      "loopStart": 0,
      "loopEnd": 8.135,
      "assets": {
        "mix": "section_A_mix.wav",
        "drums": "section_A_drums.wav",
        "bass": "section_A_bass.wav"
      }
    }
  },
  "events": [
    { "time": 0, "sectionId": "A", "bar": 1, "beat": 1, "type": "section_start" }
  ]
}
```

## Tests

Add tests for:

- manifest generation
- section durations
- loop point correctness
- rendered file existence
- stem duration alignment
- event timeline export

## Documentation

Document clearly:

- exact parity requires core-rendered assets
- native Godot procedural playback may differ
- how to choose a profile
- how new games should use generated audio kits
- how to update Godot addon when Pocket Audio Core changes

## Do not

- Do not remove existing Godot chart import.
- Do not require Godot games to run JavaScript at runtime.
- Do not claim bit-perfect native Godot procedural playback without tests.
